  1. Ingesto de dados com o Flume

    Flume um servio distribudo, confivel, disponvel para mover de forma eficiente uma grande quantidade de eventos de diferentes origens para diversos destinos. Cloudera iniciou o seu desenvolvimento e em 2011 passou a ser um projeto na ASF. Em 2012 se tornou um projeto Top-Level.

      Coleta eventos de diferentes sistemas/origens e os inserir no HDFS/HBase para centralizar os dados com escalabilidade e confiabilidade:

  1. Instalar o Apache Flume;

[hadoop@localhost ~]$ mkdir /hadoop-2.7.1/flume && tar -xzvf /install/apache-flume-1.6.0-bin.tar.gz -C /hadoop-2.7.1/flume --strip-components=1

  1. Criar um script para ingesto de dados usando o flume, considerando a origem file server e destino hdfs;

[hadoop@localhost ~]$ gedit  /hadoop-2.7.1/flume/flume_script.conf

#Flume Configuration Starts

a1.channels.fileChannel1.type = memory

# Memory Channel

# --------------------------------------------------

# capacity      100     The maximum number of events stored in the channel

# transactionCapacity   100     The maximum number of events the channel will take from a source or give to a sink per transaction

a1.channels.fileChannel1.capacity = 10000

a1.channels.fileChannel1.transactionCapacity = 10000

# Spooling Directory Source

# --------------------------------------------------

# type         The component type name, needs to be spooldir.

# fileHeader    false   Whether to add a header storing the absolute path filename.

# spoolDir             The directory from which to read files from.

# fileSuffix    .COMPLETED      Suffix to append to completely ingested files

a1.sources.source1.type = spooldir

a1.sources.source1.spoolDir = /home/hadoop/dataset/flume

a1.sources.source1.fileHeader = true

a1.sources.source1.fileSuffix = .COMPLETED

# HDFS Sink

# --------------------------------------------------

# type         The component type name, needs to be hdfs

# hdfs.path           HDFS directory path (eg hdfs://namenode/flume/webdata/)

# hdfs.batchSize        100     number of events written to file before it is flushed to HDFS

# hdfs.rollSize 1024    File size to trigger roll, in bytes (0: never roll based on file size)

# hdfs.rollInterval     30      Number of seconds to wait before rolling current file (0 = never roll based on time interval)

#hdfs.rollCount 10      Number of events written to file before it rolled (0 = never roll based on number of events)

#hdfs.writeFormat             Format for sequence file records. One of Text or Writable (the default).

#hdfs.fileType  SequenceFile    File format: currently SequenceFile, DataStream or CompressedStream (1)DataStream will not compress output file and please dont set codeC (2)CompressedStream requires set hdfs.codeC with an available codeC

#

a1.sinks.hdfs-sink1.type = hdfs

a1.sinks.hdfs-sink1.hdfs.path = hdfs://hadoop-master:9000/flume/

a1.sinks.hdfs-sink1.hdfs.filePrefix = events

a1.sinks.hdfs-sink1.hdfs.batchSize = 1000

a1.sinks.hdfs-sink1.hdfs.rollSize = 536870912

a1.sinks.hdfs-sink1.hdfs.rollInterval = 0

a1.sinks.hdfs-sink1.hdfs.rollCount = 50000000

a1.sinks.hdfs-sink1.hdfs.writeFormat = csv

a1.sinks.hdfs-sink1.hdfs.fileType = DataStream

a1.sources.source1.channels = fileChannel1

a1.sinks.hdfs-sink1.channel = fileChannel1

a1.sinks =  hdfs-sink1

a1.sources = source1

a1.channels = fileChannel1

  1. Criar um diretrio de entrada no disco local;

[hadoop@localhost ~]$ mkdir /home/hadoop/dataset/flume

  1. Criar um diretrio de sada flume no HDFS;

[hadoop@localhost ~]$ hadoop fs -mkdir /flume

  1. Executar o agent do flume para iniciar o processo de ingesto de dados;

[hadoop@localhost ~]$ /hadoop-2.7.1/flume/bin/flume-ng agent -n a1 -f /hadoop-2.7.1/flume/flume_script.conf

  1. Copiar alguns arquivos para o diretrio criado, source flume;

[hadoop@localhost ~]$  cp /home/hadoop/dataset/mapreduce/2141.xml /home/hadoop/dataset/flume

  1. Comparar o nmero de registros do arquivo de origem com o arquivo de destino;

[hadoop@localhost ~]$  wc -l /home/hadoop/dataset/flume/*

[hadoop@localhost ~]$  hadoop fs -cat /flume/* | wc -l

  1. Copiar mais arquivos para o diretrio;

cp /home/hadoop/dataset/mapreduce/2141.xml /home/hadoop/dataset/mapreduce/2141_2.xml

 2. Ingesto de dados com o Kafka

Apache Kafka um projeto message broker open-source desenvolvido pela Apache Software Foundation escrito em Scala . um sistema distribudo escalvel e durvel que recebe e envia mensagens para mltiplos produtores e consumidores em tempo real (real-time).

O Linkedin iniciou o seu desenvolvido em 2009. Em 2012, se tornou projeto Apache Incubator. Em 2014 Linkedin criou um nova empresa chamada Confluent para alavancar o projeto.

https://drive.google.com/open?id=0B-6gxa6g7uEGbWM5SVpSaWt1czA 

Producer (Produtores): Envia as mensagens para um ou mais topics dos Brokers.
Exemplos: Frontweb de aplicaes web gerando logs, proxies, servios com trace logs;

Consumer (Consumidor): Consome as mensagens de um ou mais topics dos Brokers. Utiliza um ponteiro Topic, Partition e offset(id sequencial) para controlar o progresso da leitura em um Topic. Mantm o estado de qual mensagem j foi consumida.
Exemplos:
Consumers que consomem as mensagens do Broker e as armazenam no HDFS ou DataWarehouse tradicional para anlise off-line ou processamento batch;
Consumers consumem as mensagens e as armazenam em um banco de dados NoSQL (HBase ou Cassandra) para anlise em real time / near real time.
Consomem mensagens e as armazenam em bancos de dados em memria (in-memory Redis) para gerar alertas de eventos.

Broker: Entidade que armazena as mensagens publicadas. Retm as mensagens por um determinado tempo, por exemplo: 3 ou 7 dias. Utiliza HW commidity: 2 x 4 core x86, 64 GB RAM (4 GB Kafka + 60 GB page cache), RAID 10 com 12 ou 14 discos SATA.

Zookeeper Server: Coordena a interface entre os Producers e Consumers. utilizado para detectar a adio ou remoo de brokers e Consumers, rebalancear o processamento em cada Consumer quando o evento ocorre e acompanhar qual offset foi consumido em cada partio. No controla qual consumidor leu determinada mensagem.

Mensagens: So enviadas ao Topic. Cada mensagem possui um nico offset.
Topic: onde um fluxo de mensagens de um tipo especfico armazenado. quebrado em uma ou mais parties de mensagens que so replicadas.

Partition (partio): Cada partio corresponde a um nico log lgico. Mantm a ordem das mensagens dentro da partio e garante que a mensagem no ser alterada. a menor unidade de paralelismo. Fisicamente cada log armazenado como um conjunto de arquivos com aproximadamente o mesmo tamanho, em geral 1 GB.

  1. Instalar o Kafka;

[hadoop@localhost ~]$ mkdir /hadoop-2.7.1/kafka && tar -xvf /install/kafka_2.10-0.10.0.1.tgz -C /hadoop-2.7.1/kafka --strip-components=1

  1. Validar se o Zookeeper est em execuo;

[hadoop@localhost ~]$ netstat -na | grep 2181

  1. Iniciar servio do Zookeeper;

[hadoop@localhost ~]$ /hadoop-2.7.1/kafka/bin/zookeeper-server-start.sh /hadoop-2.7.1/kafka/config/zookeeper.properties &

  1. Iniciar servio do Kafka;

[hadoop@localhost ~]$ /hadoop-2.7.1/kafka/bin/kafka-server-start.sh /hadoop-2.7.1/kafka/config/server.properties &

  1. Criar um novo tpico no Kafka;

[hadoop@localhost ~]$ /hadoop-2.7.1/kafka/bin/kafka-topics.sh --zookeeper localhost:2181 --create --topic dados1 --replication-factor 1 --partitions 2

  1. Listar o tpico criado no Kakfa;

[hadoop@localhost ~]$ /hadoop-2.7.1/kafka/bin/kafka-topics.sh --zookeeper localhost:2181 --describe --topic dados1

  1. Criar um novo producer no Kafka;

[hadoop@localhost ~]$ /hadoop-2.7.1/kafka/bin/kafka-console-producer.sh --broker-list localhost:9092 --topic dados1
