3.1 - TestDFSIO

  1. Teste de IO (Write);

O TestDFSIO ajuda na avaliação de IO do cluster, simulando um stress test, ele escreve arquivos no HDFS /benchmarks/TestDFSIO e o result é armazenado no TestDFSIO_results.log.

[hadoop@localhost ~]$  hadoop jar /hadoop-2.9.0/share/hadoop/mapreduce/hadoop-mapreduce-client-jobclient-2.9.0-tests.jar TestDFSIO -write -nrFiles 5 -fileSize 1000

  1. Teste de IO (Read);

[hadoop@localhost ~]$  hadoop jar /hadoop-2.9.0/share/hadoop/mapreduce/hadoop-mapreduce-client-jobclient-2.9.0-tests.jar TestDFSIO -read -nrFiles 5 -fileSize 1000

Throughput(N) = \frac{\sum_{i=0}^N filesize_i}{\sum_{i=0}^N time_i}

Obs: É possivel alterar o parâmetro de replicação durante os testes para comparações (TestDFSIO -D dfs.replication=1 -write ...).

  1. O comando abaixo remove o diretório criado no hdfs /benchmarks/TestDFSIO com os testes ;

[hadoop@localhost ~]$  hadoop jar /hadoop-2.9.0/share/hadoop/mapreduce/hadoop-mapreduce-client-jobclient-2.9.0-tests.jar TestDFSIO -clean

  1. O comando abaixo faz um merge nos arquivos no hdfs copiando para o disco do sistema local ;

[hadoop@localhost ~]$  hadoop fs -getmerge labfiap/* merge.txt

3.2 - TeraSort

O benchmark TeraSort é provavelmente a referência Hadoop mais conhecida. Em 2008, o Yahoo estabeleceu um novo recorde, classificando 1 TB de dados em 209 segundos - em um cluster Hadoop de 910 nodes.

Um ano mais tarde, em 2009, o Yahoo! estabeleceu um novo recorde, classificando a 1 PB (1'000 TB) de dados em 16 horas em um cluster Hadoop ainda maior de 3800 nodes (que levou apenas 62 segundos para classificar 1 TB de dados, batendo facilmente o recorde do ano anterior.

  1. Iniciando os testes do TeraSort gerando o input com o TeraGen;

* num de linhas de 100bytes

Obs: O tamanho do bloco referente ao HDFS Block Size, pode fazer uma grande diferença, trabalhando como blocos menores por exemplo 128MB, o cluster pode levar mais tempo para o start/stop de cada tarefa do que propriamente a execução. Uma boa opção seria avaliar o block size (ex: teragen -D dfs.block.size=536870912)

[hadoop@localhost ~]$  hadoop jar /hadoop-2.9.0/share/hadoop/mapreduce/hadoop-mapreduce-examples-2.9.0.jar teragen 5000000 input-teragen

  1. Rodando o TeraSort nos dados gerados;

Implementado com MapReduce sort usando chave de particionamento do range;

[hadoop@localhost ~]$  hadoop jar /hadoop-2.9.0/share/hadoop/mapreduce/hadoop-mapreduce-examples-2.9.0.jar terasort  input-teragen output-terasort

  1. Validando a ordenação dos dados de output;

[hadoop@localhost ~]$  hadoop jar /hadoop-2.9.0/share/hadoop/mapreduce/hadoop-mapreduce-examples-2.9.0.jar teravalidate output-terasort output-teravalidate

  1. Analisando o resultado;

[hadoop@localhost ~]$  mapred job -history all output-terasort

Dicas importantes

1.        Se o Job tem mais de 1TB de input considere aumentar o Block Size ;

2.        A regra geral para ajuste de memória é: use o máximo de memória que puder, sem fazer SWAP;

3.        Use compressão para o mapper output;

Ex (Terasort): map input size = output size

Block: 128MB = 1.342.177 registros (sendo 100 bytes cada registro)

Metadata: 16 (bytes) * 1.342.177 = 20.5MB

mapreduce.task.io.sort.mb = 128MB + 20.5MB = 148.5MB

  1. Alterando para usar compressão no mapper;

[hadoop@localhost ~]$ vim /hadoop/etc/hadoop/mapred-site.xml

<property>

  <name>mapreduce.map.output.compress</name>  

  <value>true</value>

</property>

<property>

  <name>mapred.map.output.compress.codec</name>  

  <value>org.apache.hadoop.io.compress.SnappyCodec</value>

</property>

  1. Dimensionando o uso de memória do YARN;

Dica: Como melhores práticas de dimensionamento podemos calcular uma base de 1-2 contêiner por disco e por core(núcleo);

Ex: 16core/16 discos/64GB RAM (8GB para o SO e 56GB para o YARN)

Abaixo especificamos a memória disponível para o nodemanager;

[hadoop@localhost ~]$ vim /hadoop/etc/hadoop/yarn-site.xml

<property>

<name>yarn.nodemanager.resource.memory-mb</name>

<value>57344</value>

</property>

Abaixo especificamos o valor mínimo de memória para cada contêiner e portanto limitando o número de contêiners:

[hadoop@localhost ~]$ vim /hadoop/etc/hadoop/yarn-site.xml

<property>

<name>yarn.scheduler.minimum-allocation-mb</name>

<value>2048</value>

</property>

  1. Dimensionando o uso de memória do Map Reduce;

Configurando Map Reduce 2 no YARN:

1) Limite de memória RAM física para cada Map e Reduce task;

No mínimo igual ou maior que a memória definida para cada alocação mínima do contêiner;

Ex:

[hadoop@localhost ~]$ vim /hadoop/etc/hadoop/mapred-site.xml

<property>

<name>mapreduce.map.memory.mb</name>

<value>4096</value>

<name>mapreduce.reduce.memory.mb</name>

<value>8192</value>

</property>

2) Limite de JVM heap size para cada task;

A recomendação é considerar 75% da memória reservada para o container(map)

ex: mapreduce.map.memory.mb * 0.75

Ex:

[hadoop@localhost ~]$ vim /hadoop/etc/hadoop/mapred-site.xml

<property>

<name>mapreduce.map.java.opts</name>

<value>-Xmx3072m</value>

<name>mapreduce.reduce.java.opts</name>

<value>-Xmx6144m</value>

</property>

3) Limite de Memória Virtual para cada task

Baseado no uso paginação, uso de memória acima do limite físico de cada contêiner.

Ex:

[hadoop@localhost ~]$ vim /hadoop/etc/hadoop/yanr-site.xml

<property>

<name>yarn.nodemanager.vmem-pmem-ratio</name>

<value>2.1</value>

</property>
