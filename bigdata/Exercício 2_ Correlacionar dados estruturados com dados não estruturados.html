Exercício 2: Correlacionar dados estruturados com dados não estruturados

Você percebe uma outra questão de negócio interessante:

Os produtos mais vistos e também os mais vendidos ?

Como o Hadoop pode armazenar dados não estruturados e semi-estruturados ao lado de dados estruturados sem remodelar um banco de dados inteiro, você também pode ingerir, armazenar e processar eventos de log da web. Vamos descobrir o que os visitantes do site realmente viram mais.

Para isso, você precisa dos dados do clickstream na web. A maneira mais comum de ingerir web clickstream é usar o Apache Flume. Flume é uma estrutura escalável em tempo real que permite rotear, filtrar, agregar e fazer "mini-operações"  carregando os dados para a plataforma de processamento escalável.

No Exercício 4, mais adiante, você pode explorar um exemplo de configuração do Flume, que será usado para ingerir e transformar em tempo real uma amostra de dados  clickstream da web. No entanto, devido ao tempo, nesta etapa, não teremos a paciência de esperar três dias de dados a serem ingeridos. Em vez disso, temos um conjunto de dados de clique da web (apenas fingir que você enviou rapidamente três dias) você pode fazer o upload em lote diretamente para o HDFS.

Dados de carregamento em lote

Temos alguns dados de registro de acesso de exemplo em /opt/examples/log_data/access.log.2.

Vamos mover esses dados do sistema de arquivos local, para HDFS.

sudo -u hdfs hadoop fs -mkdir /user/hive/warehouse/original_access_logs

sudo -u hdfs hadoop fs -copyFromLocal /opt/examples/log_files/access.log.2 /user/hive/warehouse/original_access_logs

Copie e verifique se seus dados estão no HDFS, executando o seguinte comando:

hadoop fs -ls /user/hive/warehouse/original_access_logs

Agora você pode construir uma tabela em Hive e consultar os dados via Apache Impala (incubação) e Hue. Você vai construir esta tabela em 2 etapas.

Primeiro, você aproveitará os SerDes (serializers/deserializers) flexíveis do Hive para analisar os logs em campos individuais usando uma expressão regular.

Em segundo lugar, você deverá transferir os dados desta tabela intermediária para uma que não requer nenhum SerDe especial. Uma vez que os dados estão nesta tabela, você pode consultá-la muito mais rápido e mais interativamente usando Impala.

Usaremos o aplicativo Hive Query Editor no Hue para executar as seguintes consultas:

CREATE EXTERNAL TABLE intermediate_access_logs (
   ip STRING,
   date STRING,
   method STRING,
   url STRING,
   http_version STRING,
   code1 STRING,
   code2 STRING,
   dash STRING,
   user_agent STRING)
ROW FORMAT SERDE 'org.apache.hadoop.hive.contrib.serde2.RegexSerDe'
WITH SERDEPROPERTIES (
   'input.regex' = '([^ ]*) - - \\[([^\\]]*)\\] "([^\ ]*) ([^\ ]*) ([^\ ]*)" (\\d*) (\\d*) "([^"]*)" "([^"]*)"',
   'output.format.string' = "%1$$s %2$$s %3$$s %4$$s %5$$s %6$$s %7$$s %8$$s %9$$s")
LOCATION '/user/hive/warehouse/original_access_logs';

CREATE EXTERNAL TABLE tokenized_access_logs (
   ip STRING,
   date STRING,
   method STRING,
   url STRING,
   http_version STRING,
   code1 STRING,
   code2 STRING,
   dash STRING,
   user_agent STRING)
ROW FORMAT DELIMITED FIELDS TERMINATED BY ','
LOCATION '/user/hive/warehouse/tokenized_access_logs';

ADD JAR /usr/lib/hive/lib/hive-contrib.jar;

INSERT OVERWRITE TABLE tokenized_access_logs SELECT * FROM intermediate_access_logs;

Está usando um job MapReduce, assim como nossa importação Sqoop fez, para transferir os dados de uma tabela para a outra em paralelo. Você pode acompanhar o progresso no log abaixo, e você deve ver a mensagem 'The operation has no results.' quando estiver feito.

Mais uma vez, precisamos dizer ao Impala que algumas tabelas foram criadas através de uma ferramenta diferente. Volte ao aplicativo Impala Query Editor e digite o comando:

invalidate metadata;

Agora, se você digitar 'show tables;' consultar ou atualizar a lista de tabela na coluna da esquerda, você deve ver as duas novas tabelas externas no banco de dados default. Cole a seguinte consulta no editor de consultas:

select count(*),url from tokenized_access_logs
where url like '%\/product\/%'
group by url order by count(*) desc;

Analisando os resultados você rapidamente deve perceber que esta lista contém muitos dos produtos da lista mais vendidos a partir de etapas do tutorial anterior, mas há um produto que não apareceu no resultado anterior. Um produto parece ser muito visualizado, mas nunca comprado. Por quê?

Bem, no nosso exemplo com DataPro, uma vez que esses casos estranhos são apresentados ao seu gerente, ele é imediatamente escalado. Eventualmente, alguém descobre que naquela página de exibição, onde a maioria dos visitantes parou, o caminho de vendas do produto teve um erro de digitação no preço do item. Uma vez que o erro de digitação foi corrigido, e um preço correto foi exibido, as vendas para esse SKU começaram a aumentar rapidamente.

Conclusão:

Se você não tivesse uma ferramenta eficiente e interativa que permitisse análises em dados semi-estruturados de alto volume, essa queda de receita teria sido mantida por muito tempo. Há risco de perda se uma organização procura respostas dentro de dados parciais. Correlacionar dois conjuntos de dados para a mesma pergunta comercial mostrou valor, e será capaz de construir dentro da mesma plataforma facilitando a vida para você e para a organização.
